01Executive summary

Confirmed: uv run pytest -q passed 109 tests (1 skipped); uv run pytest -q collab/tests passed 452 tests (6 skipped, 23 subtests); dashboard unit tests, Oxlint, and the Next production build passed. uv lock --check also passed.

Confirmed failures: the configured Pyright check cannot start; the README's repo-wide Ruff command exits red with 321 findings; dashboard formatting exits red for all 17 tracked files because Oxfmt has no project config; and status documents still describe a pre-code workspace.

Important distinction: this scan did not find a failing trading-core test or a failing dashboard build. The project feels broken because its evidence and delivery surfaces disagree about what has been checked and what currently exists.

06Findings by lane

  • Behavior: Python core, collab, and dashboard tests passed. No failing assertion reproduced.
  • Static: core scope is Ruff-clean; whole-repo Ruff is not. The failure distribution is predominantly line length rather than one functional rule.
  • Build: dashboard production build passed; Python lock consistency passed. Pyright cannot be invoked.
  • Docs/control plane: roadmap and Kickoff lag the checked-out implementation.
  • Safety: gateway/assembler ownership was checked and appears aligned with the sealed-context design; the known Pydantic provenance bypass remains a release boundary caveat.

07Cross-lane duplicate groups

verification-topology merges the red root Ruff command, uncollected collab tests, missing runnable Pyright, and absence of an authoritative aggregate check. Dashboard formatting remains separate because its direct root cause is a missing Oxfmt policy, even though it contributes to the same user-visible feeling of broken checks.

08Human-decision blockers

INT-005: hardening or replacing mint guards changes a security/production-safety boundary and may require a different architecture. The audit intentionally does not select it. Separately, choosing whether aggregate verification must be CI-enforced rather than local-only is a project-process decision; the report recommends an aggregate local command first, but does not assume CI authorization.

09Quick wins

INT-001 is the recommended first candidate: small blast radius, a deterministic red loop, and a clear expected green state. INT-003 is also contained, but formatter-policy choices should be checked before reformatting every dashboard file.

10High-leverage deep work

INT-002 is the durable repair: make each supported subsystem's ownership and checks explicit, then compose them in one authoritative command. It prevents future “green test suite / red project” ambiguity, but touches Python configuration, collab conventions, dashboard scripts, documentation, and potentially CI.

11Unsafe-to-auto-execute work

Do not auto-execute INT-005. The correct fix may involve package export policy, trusted process boundaries, adapter behavior, or a different enforcement model; this is a safety architecture decision, not a lint-style patch.
