4 · Observability — the finding that changes the design [verified]

Seat stderr is dropped on success. autopilot.py:522-594 (_cli_runner, used by driver rounds and lanes alike) captures stderr to a TemporaryFile, kills the seat past 64 KiB, and on exit 0 never reads it again; on failure only 500 chars survive into telemetry. Therefore stderr usage lines are a dev/pytest aid only — not run telemetry.
Fork 1 of 1 — where does durable cache telemetry live?

My lean — (A) adapter-side JSONL side-channel: adapters append one line per POST (ts, adapter, api, model, step, prompt_tokens, cached_tokens, cache_write_tokens?) to a file named by env COLLAB_USAGE_LOG (loaded from collab/.env, which the adapters already read; unset → stderr only). Adapters-only change, no driver surgery, works for manual runs too, hermetic-testable with tmp_path.

Counter-argument — (B) driver persists stderr tail: put the 64 KiB tail into the round record in _cli_runner, making ALL seat stderr durable (useful beyond caching) and dashboard-surfaceable. Costs a driver/lanes touch (higher blast radius: every seat execution path), and 500-char failure snippets already exist there. (A) and (B) are not exclusive; (B) can be its own later handoff.

Tests (hermetic — existing pattern: monkeypatch _post_json, SEAT_ENV_FILE → nonexistent)

  • Fake post records every (url, payload); multi-step script (tool-call response, then final). Assert prompt_cache_key present and identical in every captured payload, including a max_steps=1 exhaustion case covering the two budget-final sites.
  • Fake usage objects in both API shapes (+ one with usage absent) → assert one stderr line per POST with the right step ordinal and cached-token count; absent-usage case logs "not reported" and exits 0.
  • Chat replay test: response carrying reasoning_content → next request's assistant message includes it; response without → key absent.
  • JSONL side-channel (if Fork = A): COLLAB_USAGE_LOG=tmp_path → valid JSONL, one record per POST; unset → file untouched.
