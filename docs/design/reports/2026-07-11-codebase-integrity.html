1. Executive summary

Current state: the in-progress PT-4a domain contract is internally tested, but it is not integrated with the existing IBKR snapshot path. Every exercised gateway snapshot attempts the now-forbidden direct construction of RiskContext, so that path is deterministically red.

This is not evidence that PT-4a violated its own handoff: the current handoff explicitly says “No consumers wired here.” It is evidence that the whole checkout is not integration-ready until the planned AccountSnapshot/assembler consumer work lands. The safest next unit is that consumer seam, not a one-line authority bypass.

A separate, contained build-integrity issue is confirmed: the repository configures Pyright but does not declare it in the dev dependency group, and the active environment cannot run python -m pyright. Ruff also reports 301 findings when the changed collab source and tests are included; that is recorded as an intentionally rejected broad-cleanup opportunity rather than folded into this loop.

5. Candidate ledger

INT-001 — Complete the sealed RiskContext consumer/assembler seam

Wiring / Integrationdesignconnected-impact-sweep

Current friction: the gateway’s build_risk_context still passes symbol-keyed positions into a direct RiskContext(...) constructor. PT-4a makes that type assembler-minted, conId-keyed, holdings-based, and requires additional causal metadata. Snapshot construction therefore raises before any paper-account context is returned.

Root seam: PT-3 IBKR adapter / planned AccountSnapshot + decision-context assembler / sealed PT-4a domain contract.

Integrity rule: a consumer may create a sealed causal snapshot only through its designated assembler; it must not retain a parallel legacy shape or bypass authority.

  • src/ibkr_trader/domain/models.py:160-207 defines the guarded, holdings-based RiskContext; :148 rejects direct construction.
  • src/ibkr_trader/ibkr/gateway.py:152-180 constructs the old symbol-keyed shape directly.
  • tests/test_ibkr_gateway.py:86, 94, 104, 178, 293, 339, 363, 376, 392, 409 fail with the same TypeError.
  • handoffs/claimed/029-pt-4a-domain-riskcontext-holdings.md:57 deliberately defers consumer wiring; this is therefore a checkout-level integration finding, not a claim that PT-4a must shortcut its own scope.

Confirmed facts: the failure is deterministic; focused PT-4a tests pass; the existing gateway cannot populate the new contract. Plausible inference: the planned PT-4b AccountSnapshot and PT-4c assembler handoffs are the intended owning implementation. Unknown: whether either handoff’s current draft fully defines the migration of legacy test fixtures and fake broker values.

Scores: severity 4, confidence 5, leverage 5, locality 2, testability 5, blast radius 5, regression risk 3, human-decision risk 2. Priority: 11.

Enforceable check: a fake-gateway → AccountSnapshot → assembler integration test that proves a valid broker response produces a sealed, conId-keyed context and that incomplete identity/mark data fails closed.

Verification: add the seam test, run pytest tests/test_ibkr_gateway.py tests/test_domain_models.py with a writable temp root, then the full root suite plus Ruff and Pyright. Adjacent concerns out of scope: PT-5 rules, execution control, and closing the provenance-bypass release gate.

INT-003 — Specify the release gate that closes mint-provenance bypasses

Architecture fitness / Securitydesignblocked-needs-human-decision

Current friction: RiskContext.model_construct() bypasses the constructor guard and the authority remains importable from the implementation module. The code and ADR accurately label this as provenance rather than an in-process security boundary, but the project has not yet chosen the executable release-gate control that makes this acceptable before live use.

Root seam: Python/Pydantic construction model ↔ authority-containment policy ↔ live-release governance.

Integrity rule: safety claims must match their actual enforcement boundary; a live-release condition cannot remain merely documented.

  • src/ibkr_trader/domain/models.py:140-160 states the guard is not an in-process security boundary.
  • tests/test_domain_models.py:158-161 proves the model_construct() bypass.
  • docs/design/adr/0003-pt5-rules-ledger.md:249-252, 341 defers bypass closure to a release gate before live use.

Confirmed facts: the bypass exists and is intentionally documented. Plausible inference: an executable release gate is safer than a prose-only reminder. Unknown: whether the gate should be a static import rule, a packaging boundary, a capability-injection policy, an adversarial test suite, or a combination.

Scores: severity 3, confidence 5, leverage 4, locality 2, testability 4, blast radius 4, regression risk 4, human-decision risk 5. Priority: 5.

Blocked by: the security/trust-boundary and live-release-policy decision. Enforceable check opportunity: selected only after that policy is decided. Adjacent concerns out of scope: implementing PT-4b/PT-4c and changing the present provenance claim.

6. Per-category findings

Wiring / integration

The only non-test direct constructor search hit outside the domain test is gateway.py:180. This makes the break narrow and measurable, but it is not a valid direct-authority patch because the new contract also changes identity, holdings, mark provenance, and digest fields.

Build / static

Pyright is configured but not present. Ruff runs, but ruff check src tests collab/tools/lib collab/tests reports 301 findings (31 mechanically fixable). Those findings span many existing collab files, including tests and unrelated runtime helpers. This audit intentionally rejects a broad “fix all lint” cleanup: select a narrow lint-scope/ownership candidate later if the repository wants the collab tree to share the root quality bar.

Test environment

Tests using tmp_path could not create or scan the current user pytest temp directory because of WinError 5. This is an audit-environment limitation, not yet a repository defect. It prevented complete validation of state, reconciliation, and collab tests; it does not explain the 10 direct-constructor failures.

8. Human-decision blockers

INT-003: choose the authority-containment / release-gate model before asking an implementation to close a documented bypass. This is a security-boundary and release-policy decision; auto-execution would invent a trust model.

INT-001: no human decision is currently required to acknowledge the red seam, but its execution must follow the selected PT-4b/PT-4c contract rather than shortcut the staged migration.

9. Quick wins

INT-002 is the only high-locality, high-testability candidate. Declaring the configured checker is small, independently verifiable, and does not alter trading behavior. It should not be bundled with any domain migration work.

10. High-leverage deep work

INT-001 is high-leverage because a correctly assembled causal context is the input to every later risk decision. Its larger blast radius is appropriate: identity resolution, holdings valuation, price provenance, fake broker fixtures, gateway tests, and the mint boundary must agree.

11. Unsafe-to-auto-execute work

INT-003 is unsafe to auto-execute due to human-decision risk 5. Any chosen policy must preserve the current honest distinction between provenance and security, and be reviewed before live execution.
